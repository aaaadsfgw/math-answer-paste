# 開発ログ

## 初期設計

Chrome拡張 Manifest V3 として、ポップアップ中心の小さな学習支援ツールとして設計しました。UIは青・グレー系で、提出課題として見やすいカード型レイアウトにしています。

## ローカルLLM採用理由

APIキーを使うクラウドAIではなく、OllamaによるローカルLLMを採用しました。問題文を外部サーバーへ送らず、ユーザーのPC上で完結させるためです。

## 実装方針

Ollama通信、プロンプト生成、回答抽出、デモ回答、保存処理を別ファイルに分けました。AIが返す `<think>...</think>` は表示用の最終結果から取り除きます。

## モデル比較結果

RTX 4060 VRAM 8GB 環境で実機テストを行いました。

- `deepseek-r1:14b`: 正答できるがCPU/GPU混在でやや重い
- `deepseek-r1:8b`: GPU 100%でも回答が遅かった
- `qwen3:8b`: 回答精度と速度のバランスが良いため初期採用

この結果を受けて、初期モデルを `deepseek-r1:14b` から `qwen3:8b` に変更しました。

## 出力形式の変更

最初は検証しやすさを優先して、AIの最終回答を `[ ]` 形式で扱っていました。しかし実際のコピーしやすさを考えると、最終仕様では `x=4` や `ウ` のような答え単体のみをコピーできる形が適しています。

そのため、表示・コピー対象を角括弧なしの「答えだけ1行」に変更しました。AIが誤って `[x=4]` のように返した場合も、アプリ側で外側の角括弧を外します。

## 会話履歴を送らない設計

CLIでは前の問題に引っ張られることがあるため `/clear` が必要になる場合があります。アプリではAPI呼び出しごとに現在の `system` と `user` だけを `messages` に入れ、過去の会話履歴を送らない設計にしました。

## /no_think の採用

`qwen3:8b` では user message の先頭に `/no_think` を入れることで応答速度の改善が見られました。そのため標準プロンプトに `/no_think` を採用しました。

## 今後の確認

Chromeの拡張管理画面で読み込み、デモモードの固定例、OllamaモードのAPI疎通、履歴保存、設定保存を確認します。
